# #132 — drop axios: the main entry's last "optional" peer that wasn't

## Problem

`#142` split the chart components to a `/charts` entry, but the main entry
still statically imports **axios**, so the "optional" peer is mandatory for
every consumer (`import qd from "axios"` sits at the top of
`dist/dashboard-for-laravel.js`). A consumer without axios installed fails to
build — the same failure custard #104 hit for chart.js.

## Finding: no subpath needed

The issue suggested mirroring the `/charts` split (a `/table` entry). But the
library only uses axios in **two plain GETs**:

- `DXTable.vue` `internalProvider` — `axios.get(props.apiUrl, { params })`
- `useResourceEditor.ts` `fetchFullRecordForEdit` — `axios.get(showUrl)`

Everything else (all of `useForm`/`defineForm`, i.e. every POST/PUT/PATCH/
DELETE the library makes) already goes through the library's own fetch-based
client, **`resources/js/utils/api.ts`** — which is exported from the index and
already provides:

- the same Laravel bracket-notation query serialisation axios produces
  (`filters[status]=active`, `filterValues[]=…`) via `toQuery`;
- a `{ data }` response shape, so call sites keep reading `.data`;
- *better* error messages (`ApiError.message` with status-specific text) than
  the axios branch (`error.response.data.message`);
- the consumer config surface axios interceptors were the workaround for:
  `api.setBaseURL()` / `api.setDefaultHeader()`, plus CSRF + `Accept:
  application/json` + `X-Requested-With` headers on every request.

So the right fix is: **route the two GETs through `api.get` and remove axios
entirely** — no subpath, no split, one HTTP stack instead of two.

## Behaviour notes

- A consumer whose axios interceptors decorated these two GETs (auth header,
  baseURL) loses that — but such a consumer was already broken for every form
  submission (fetch-based since the beginning), and `api.setDefaultHeader` /
  `setBaseURL` is the sanctioned mechanism. Recorded in CHANGELOG.
- These GETs now also send `X-CSRF-TOKEN` / `X-Requested-With` (harmless on
  GET, consistent with the rest of the library).
- Error shape reaching `wrappedProvider` changes from axios's
  `error.response.data.message` to `ApiError`'s `error.message`; the handler
  already falls back to `error.message`, and the axios-shape branch stays for
  **consumer-supplied providers** that may still use axios themselves.

## Steps

1. **Red-first guard**: extend `tests/bundle/chart-optional-peer.test.ts`
   pattern with an axios assertion (main bundle must not reference `"axios"`);
   watch it fail against the current dist.
2. Swap the two call sites to `api.get` (params as second arg).
3. Tests: `vi.spyOn(api, 'get')` instead of axios; mock signature `(url,
   params)` not `(url, { params })`; rejection fixtures become `ApiError`
   shape (`{ message, errors, status }`) so they exercise the real path.
4. Remove axios from `peerDependencies` + `peerDependenciesMeta` and from
   `vite.config.ts` externals/globals.
5. Docs: DXTable.mdx "via axios" wording → the built-in fetch client.
6. CHANGELOG entry (minor: consumers read this to drop the peer / migrate any
   interceptor usage to `api` config).

## Outcome

Landed as planned — no subpath, axios removed outright. The two GETs now use
`api.get`; axios is gone from `peerDependencies`, `peerDependenciesMeta`, and
the vite externals/globals; DXTable.mdx wording updated; CHANGELOG (Unreleased
→ next minor) documents the peer removal + the interceptor→`api` migration.
All 503 tests green (502 existing + the new bundle guard), `vue-tsc` clean.

The guard (`tests/bundle/chart-optional-peer.test.ts`, axios describe) was
verified red twice: once against the pre-fix dist as intended, and once more
when the "fixed" suite STILL failed — which exposed a real machine-level trap:
`~/.npmrc ignore-scripts=true` makes npm skip `pre`/`post` hooks on `npm run`,
so `pretest:headless` never rebuilt `dist/` and the suite was testing the
stale bundle. Local mitigation: run `npm run build:lib` explicitly before
tests; `release.sh` now does exactly that before its test gate (its bundle
guards previously certified the PREVIOUS release's dist). CI unaffected.

Codex review (post-implementation) found one real P1 + one P2, both fixed:

- **P1: `api.get` double-`?`** — an `apiUrl` already carrying a query string
  (`/api/things?filter[archived]=1`, used verbatim in the DXTable suite) got
  params appended with a second `?`, corrupting the query. Masked in tests
  because the suite mocks `api.get`. Fixed with `?`/`&` merge; regression
  tests at the fetch level in `tests/utils/api-get.test.ts` (verified red).
- **P2: non-simple headers on GET** — the client sent `Content-Type` and
  `X-CSRF-TOKEN` on GETs (axios didn't), forcing a CORS preflight on
  cross-origin endpoints. Bodyless GET/HEAD now skips both; `X-Requested-With`
  and `Accept` stay (Laravel convention, matches all existing form traffic).

Final: 509 tests green, typecheck clean.
