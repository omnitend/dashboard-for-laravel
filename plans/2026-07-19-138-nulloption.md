# #138 — DAutocomplete `null-option` (searchable single-select convergence)

## Context

#138 wanted a searchable single-select that takes the best of `DAutocomplete`
and the house `ot-searchable-select`. After #153 was verified fixed upstream
(numeric values already round-trip as numbers on bvn 0.45.6), the target spec
collapses to **one** real gap:

| #138 spec item | status |
| --- | --- |
| Combobox keyboard model (one tab stop, type-ahead, arrows, Enter, Esc) | ✓ already (BAutocomplete) |
| Identity-preserving value (number stays a number) | ✓ confirmed (#153, guarded) |
| No re-point-on-type wart | ✓ DAutocomplete never did this |
| **Explicit null / placeholder option** | ✗ **this plan** |

So #137's wrapper-strategy question resolves to "keep and extend `DAutocomplete`"
(the issue's preferred path (a): a mode on DAutocomplete, not a new component).

## The gap

The house component pins a `value: null` option that is **visible, never
filtered, and selectable** so "no selection" is always reachable in-list.
`DAutocomplete` has no such thing — the placeholder role falls to the
`placeholder` prop + the ✕ clear button only.

## Design

Add one opt-in prop on `DAutocomplete`:

- `null-option?: boolean | string`
  - falsy / absent → off (unchanged behaviour)
  - `true` → on, default label `"None"`
  - `string` → on, that string as the label

When on, the wrapper:
1. Prepends `{ value: null, text: <label> }` to the consumer's `options`.
2. Supplies a `filterFunction` that **always keeps the null option** (pinned to
   the top, never filtered by the search term) and applies the consumer's
   `filterFunction` — or bvn's default substring-on-text — to the rest.

Selecting it sets `modelValue = null` → bvn's `hasSelection` reads null as "no
selection" → ✕ hidden, placeholder shows. That is the desired cleared state, and
matches the issue's `value: null` semantic.

### Why null (not `''` or a sentinel)

bvn drops the **entire** option list if any option value is `''` (guarded
upstream trap in `DAutocomplete.test.ts`), and `ComboboxItem` throws on `''`.
`null` is accepted, and is what the issue specifies. DXTable does **not** use
this prop (it builds its own "All" option with a sentinel), so the `''` trap is
untouched.

### Wrapper mechanics

Declare `options`, `filterFunction`, `nullOption` as real props so they leave
`$attrs`; everything else keeps flowing through `v-bind="$attrs"`. Bind
`:options="computedOptions"` and `:filter-function="effectiveFilter"`
explicitly. The existing clear-button divergence (reads `modelValue` from attrs)
is untouched.

## Tests (must fail against unfixed code)

- null option renders and is **pinned** even when the search term matches none of
  its text (type a term that filters out everything else — null option remains).
- selecting the null option emits `null` and hides the ✕.
- a consumer `filterFunction` still runs on the non-null options (compose, don't
  clobber).
- prop off → no null option, list unchanged (regression guard).
- typecheck: `null-option` accepts boolean and string.

## Out of scope

- `multiple`/tags null handling (single is the core case per #138).
- Any change to DXTable's filter "All" option.
- The broader #137 raw-alias refactor (separate major).

## Outcome

Done, 2026-07-19.

- #153 verified fixed-upstream and closed; guard added
  (`tests/components/DAutocomplete-value-roundtrip.test.ts`, 3 tests, commit
  c1aba66) — value identity is real on bvn 0.45.6, no wrapper fix was needed.
- `null-option` prop added to `DAutocomplete` (`boolean | string`): prepends a
  pinned `{ value: null, text }` row and supplies a composed `filterFunction`
  that keeps it immune to the search term while running the consumer's filter
  (or bvn's default) on the rest. Types aligned to bvn's `SelectOption` /
  options signature (`vue-tsc` clean).
- Tests: `tests/components/DAutocomplete-null-option.test.ts` (6 tests) — off by
  default, pinned render, custom label, pinned-under-empty-filter, select→null +
  ✕ hidden, and filterFunction composition. Verified 5/6 go red against the
  unfixed component (the "off by default" one correctly stays green). Full suite
  475 passing; docs build green.
- Docs: `DAutocompleteExample.vue` gains a numeric-id null-option demo with a
  live `typeof` readout; `DAutocomplete.mdx` documents the extra prop + value
  round-trip.

Not a DIVERGENCES.md entry: `null-option` is an **addition**, not a shield over
a changed bvn signature.

Downstream: greendragon can converge its three searchable-select usages onto
`DAutocomplete` + `null-option` and drop the `String()`/`Number()` workaround
(tracked app-side in omnitend/omnitend#647).
